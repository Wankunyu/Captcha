## Implementation Decisions

### Dataset Expansion Strategy

- **D-01:** Use a hybrid expansion strategy: supplement original low-sample categories and add exactly two new static-compatible CAPTCHA categories.
- **D-02:** Supplemented original categories should prioritize low-sample, hard, underpowered, threshold-sensitive, or claim-relevant task families rather than easy categories that add little evidentiary value.
- **D-03:** The two new categories must be static-compatible: offline dataset instances with ground truth and evaluable answer formats. They must not require live service interaction, temporal hold behavior, browser automation, drag/slider execution, or production endpoint access.
- **D-04:** New categories may use clear local task-language labels rather than matching the original CaptchaWorld taxonomy exactly, but they must record the mapping rationale and structural feature being tested.

### Dataset Placement and Provenance

- **D-05:** Store expanded dataset files in a sidecar expanded-dataset directory with manifest-driven provenance. Do not directly merge expanded samples into the original `captcha_data/` directory.
- **D-06:** The manifest must preserve evidence separation among original CaptchaWorld evidence, supplemented-category evidence, and new-category evidence.
- **D-07:** Planning should reuse or extend the Phase 3 extended-dataset manifest pattern rather than inventing a notebook-only data ledger.
- **D-08:** Every expanded row should record source id, source path, task type, task family, sample count, label format, metadata alignment notes, answer-format normalization, compatibility status, evaluation status, limitation notes, and whether it is eligible for adaptive evaluation.

### Supplemental Experiment Scope

- **D-09:** Run the expanded dataset on all models in the existing paper-facing experiment matrix. Do not reduce this phase to only the strongest model or a single cross-provider baseline.
- **D-10:** The static supplemental run should follow the paper-facing optimized-prompt/Exp2-style validation path unless planning identifies a stricter paper-aligned default already used in the current result matrix.
- **D-11:** Include an Adaptive Attacker Exp3-style supplemental run in this phase.
- **D-12:** The adaptive supplemental run must preserve the Phase 2 threat model: offline dataset instances, binary pass/fail feedback only, explicit local policy-memory notes, no ground-truth labels, no coordinates/counts/categories as corrective hints, and append-only adaptive attempt records.
- **D-13:** Because this is a full supplemental experiment phase, planning must include request-count and cost previews before execution. Provider runs must be gated by preflight, unique or resumable output directories, run manifests, attempt logs, and secret-safe reporting.
- **D-14:** "All models" means the existing paper-facing model matrix already used by the COGNITION revision evidence, not an open-ended invitation to add unrelated new providers or experimental models.

### Paper Integration and Claim Boundaries

- **D-15:** Use Phase 04.1 results to upgrade the main paper evidence, not only appendix material or reviewer-response artifacts.
- **D-16:** Add expanded-dataset validation analysis to the relevant main-body section and update main tables/claims where the results support or constrain them.
- **D-17:** If expanded data diverges from original conclusions, the paper must surface the divergence explicitly instead of smoothing it into the original claim.
- **D-18:** Expanded-dataset evidence should strengthen benchmark breadth and structural-boundary claims, but it must not erase CaptchaWorld limitations or imply population-level deployment estimates.
- **D-19:** The final prose should distinguish contextual external SOTA baseline triangulation from directly evaluated expanded-dataset evidence.

### the agent's Discretion

- The planner may choose exact sidecar directory names, manifest filenames, schema class names, and CLI subcommand names, provided the provenance and evidence-separation decisions above are preserved.
- The planner may choose the exact low-sample original categories after inspecting current dataset counts and claim relevance.
- The planner may choose the two new static-compatible categories after feasibility research, but both must remain offline and evaluator-compatible or come with explicit normalization/scoring tasks.
- The planner may choose whether to extend existing Phase 3 manifest schemas or introduce Phase 04.1-specific schemas, provided strict validation and CSV/JSON outputs remain.
- The planner may decide the exact compact main-paper table shape, as long as expanded-dataset evidence is visible in the main text and claim boundaries are explicit.

## Common Pitfalls

### Pitfall 1: Manifest Rows That Cannot Be Evaluated

**What goes wrong:** A manifest can describe a new category, but `run_eval.build_tasks()` will skip it unless the task type is in `SUPPORTED_TYPES` and has an expected directory, prompt, schema, and scorer. [VERIFIED: run_eval.py]

**Why it happens:** The current evaluator uses task-type switches and filesystem conventions rather than a dynamic task registry. [VERIFIED: run_eval.py]

**How to avoid:** Add compatibility tests that call `build_tasks()`, `build_json_schema()`, and `evaluate_pass1()` for every Phase 04.1 task before provider runs. [VERIFIED: tests/test_task_contracts.py; VERIFIED: tests/test_scoring_regressions.py]

**Warning signs:** Preflight passes a manifest row but selected task count is zero or the task is absent from static/adaptive summaries. [VERIFIED: revision_preflight.py; VERIFIED: adaptive_preflight.py]

### Pitfall 2: Sidecar Layout Does Not Match Evaluator Layout

**What goes wrong:** Source files are stored under provenance folders, but evaluator code expects `dataset_root/<TaskType>/ground_truth.json`. [VERIFIED: run_eval.py]

**Why it happens:** Phase 04.1 needs source provenance and evaluator compatibility at the same time. [VERIFIED: 04.1-CONTEXT.md]

**How to avoid:** Materialize `evaluator_slice/` from the manifest and run static/adaptive preflight against that root. [VERIFIED: run_eval.py; VERIFIED: revision_preflight.py; VERIFIED: adaptive_preflight.py]

**Warning signs:** `revision_preflight.py` raises missing `ground_truth.json` or reports unexpected selected counts. [VERIFIED: revision_preflight.py]

### Pitfall 3: "All Models" Drifts Into New Provider Scope

**What goes wrong:** The phase accidentally adds experimental providers or omits an existing paper-facing model. [VERIFIED: 04.1-CONTEXT.md]

**Why it happens:** `visualize_results.DEFAULT_MODEL_NAMES` contains display names, but actual existing Exp2 result directories define the current paper-facing matrix. [VERIFIED: visualize_results.py; VERIFIED: results/exp2 directories]

**How to avoid:** Generate an explicit `expanded_run_matrix.csv/json` from current paper-facing result directories and require user confirmation before adding any new provider/model. [VERIFIED: results/exp2 directories; VERIFIED: 04.1-CONTEXT.md]

**Warning signs:** Run matrix includes a provider/model not present in current Exp2 results or excludes a current Exp2 provider/model without a caveat. [VERIFIED: results/exp2 directories]

### Pitfall 4: Adaptive Run Accidentally Leaks Labels or Hints

**What goes wrong:** Reflection or memory stores answers, coordinates, counts, labels, puzzle IDs, or corrective hints. [VERIFIED: adaptive_artifacts.py; VERIFIED: adaptive_attacker.py]

**Why it happens:** Adaptive prompts use the model's raw/parsed answer and binary fail feedback, so tests must guard against adding ground truth into helper signatures or policy notes. [VERIFIED: tests/test_adaptive_attacker.py]

**How to avoid:** Reuse `build_reflection_prompt()`, `AdaptivePolicyState`, and banned-token/banned-pattern validators, and add Phase 04.1 tests with sentinel ground-truth values. [VERIFIED: adaptive_attacker.py; VERIFIED: adaptive_artifacts.py; VERIFIED: tests/test_adaptive_end_to_end.py]

**Warning signs:** `adaptive_attempts.jsonl` contains ground-truth sentinel strings or policy-note validation rejects rows. [VERIFIED: tests/test_adaptive_end_to_end.py; VERIFIED: adaptive_artifacts.py]

### Pitfall 5: Divergence Gets Smoothed Into Positive Evidence

**What goes wrong:** Expanded data contradicts original conclusions but the paper table still reports only a strengthened boundary. [VERIFIED: 04.1-CONTEXT.md]

**Why it happens:** Existing comparison artifacts can classify agreement/divergence, but Phase 04.1 must apply that logic across all model rows and evidence origins. [VERIFIED: extended_dataset_manifest.py]

**How to avoid:** Require `agreement_status`, `divergence_reason`, and `claim_boundary_note` fields in paper-ready rows. [VERIFIED: extended_dataset_manifest.py; VERIFIED: 04.1-CONTEXT.md]

**Warning signs:** A row crosses the 40% cutoff direction but has no divergence note. [VERIFIED: extended_dataset_manifest.py; VERIFIED: statistical_confidence.py]

### Pitfall 6: Provider/Protocol Failures Become Scientific Evidence

**What goes wrong:** Timeouts, malformed responses, parser failures, or provider exceptions are counted as structural CAPTCHA robustness evidence. [VERIFIED: adaptive_artifacts.py; VERIFIED: failure_taxonomy.py]

**Why it happens:** Legacy `run_eval` aggregate CSVs focus on pass rates, while adaptive artifacts carry explicit failure classes. [VERIFIED: run_eval.py; VERIFIED: adaptive_artifacts.py]

**How to avoid:** Feed static and adaptive supplemental outputs into failure-taxonomy-style rows before claim updates. [VERIFIED: failure_taxonomy.py; VERIFIED: 03-CONTEXT.md]

**Warning signs:** Paper-ready rows lack scientific/protocol/infrastructure failure counts or caveats. [VERIFIED: failure_taxonomy.py]
