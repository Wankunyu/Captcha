# Phase 04.2 Source Candidate Notes

Date: 2026-05-21

Purpose: record real external CAPTCHA/source categories that are worth keeping in the Phase 04.2 candidate pool, including useful alternatives beyond the current `Symbol_Count` and `Relation_Match` replacement search. These notes are advisory only; a category still needs manifest staging, license/provenance validation, and offline evaluator compatibility before it can appear in the selected manifest.

## Current Best Fits

| `Symbol_Count` | `Occluded_Pattern_Counting` | NextGen-CAPTCHAs | 50 | Best current fit. Static image asks the solver to count specified color/shape symbols; ground truth includes target colors, shapes, and counts. | Original answer is usually two counts such as `count1,count2`. Prefer deriving a single-target `Symbol_Count` row from `metadata.target1_*`, or explicitly extending scoring for paired counts if the phase plan approves that contract change. |
| `Relation_Match` | `geetest` | VIPER_code Zenodo record | 1000 question rows and 1000 detection rows | Best current fit. Detection metadata includes candidate objects and `ans_id`. | Materialize offline reference/context image plus object-crop options; map `ans_id` to `correct_index`. This is a schema adaptation, not a live CAPTCHA workflow. |

## NextGen-CAPTCHAs Categories Worth Keeping

NextGen-CAPTCHAs has an MIT license in the repository. Its `captcha_data` snapshot is useful for future hard-category expansion, but most G3/counting-family entries are grid-select tasks rather than drop-in `Symbol_Count` rows.

| Category | Observed count | Answer shape | Why keep it | Phase 04.2 suitability |
|----------|----------------|--------------|-------------|------------------------|
| `Occluded_Pattern_Counting` | 50 | Numeric string, usually paired counts | Strongest symbol-count replacement. Counts color/shape symbols under occlusion. | High, with single-target normalization or approved paired-count scoring. |
| `Hole_Counting` | 20 | Grid-select indices | Counts holes in shapes; good structural numerosity signal. | Medium. Useful as a future `grid_select` counting variant, not a drop-in `Symbol_Count`. |
| `Color_Counting` | 20 | Grid-select indices | Requires color-count reasoning over sketches. | Medium. Name is tempting, but output is cell selection, not numeric count. |
| `Layered_Stack` | 20 | Grid-select indices | Combines occlusion, layering, and exact lower-layer counts. | Medium. Good future hard structural-count category. |
| `Illusory_Ribbons` | 20 | Grid-select indices | Topological loop counting under visual ambiguity. | Medium. Good future hard-count/topology category. |
| `Subway_Paths` | 20 | Grid-select indices | Route-counting with rule constraints. | Medium. Useful for reasoning-depth claims, but not current task contract. |
| `Spooky_Circle_Grid` | 20 | Grid-select indices | Simple visual category selection with animation/noise style. | Low to medium. Useful for robustness variety, less close to current target needs. |
| `Dice_Roll_Path` | 20 | Final die top value | Procedural spatial/dice reasoning, distinct from `Dice_Count`. | Low for current 04.2, but useful for future reasoning-heavy dice tasks. |
| `Rotation_Match` | 20 | Grid-select indices | Rotation-invariant shape matching; could help if hard-family scope later reopens `Rotation_Match`. | Medium for future hard-family expansion; not a current 04.2 target. |
| `Backmost_Layer` | 20 | Grid-select indices | Layer/occlusion relation reasoning. | Medium for future relation/spatial expansion. |

## VIPER_code / Zenodo Categories Worth Keeping

The Zenodo record is `VIPER_code`, DOI `10.5281/zenodo.18191465`, version v2, published 2026-01-09, licensed CC-BY-4.0. It is an attack-framework artifact, so only offline dataset metadata/images should be used. Do not run or integrate its live attack framework.

| Category | Observed count | Why keep it | Phase 04.2 suitability |
|----------|----------------|-------------|------------------------|
| `geetest` | 1000 questions, 1000 detection rows | Strong spatial/semantic relation prompts with object detections and answer IDs. | High for `Relation_Match` if materialized as reference/context plus object-crop options. |
| `netease` | 1000 questions, 1000 detection rows | Many orientation/same-direction prompts over characters and objects. | Medium to high as a fallback relation source; narrower semantics than `geetest`. |
| `VTT_dataset` | 1093 images, 1992 question text files | Many orientation, above/below, and uppercase/lowercase correspondence prompts. | Medium. Useful relation source, but question/image/label alignment needs cleaning. |
| `shumei` | 270 question rows | Size/color/3D-object target prompts. | Medium for future attribute-click or geometry-click variants; weak for option-based relation matching. |
| `xiaodun` | 1000 question rows | Mixed color/category/letter/object prompts, with some count-like and attribute prompts. | Medium for future object/attribute click expansion; lower fit for current `Relation_Match`. |
| `dingxiang` | 1000 question rows | Plane/solid/letter-case/color prompts, lower relation density. | Low to medium; useful if future work needs attribute-click variety. |

## Staged Source Decision

The Phase 04.2 sidecar now stages real external sources only; no GPT Image fallback instances are needed. OpenCaptchaWorld additions are selected by cleaned-ground-truth key novelty, not image hash novelty. If a puzzle id is absent from the local cleaned `ground_truth.json`, it is treated as previously unused even when the same image bytes already exist under `captcha_data`; exact hash matches are retained as review warnings in the validation artifacts.

Staged counts:

| Task type | Source category | Staged samples | Source total | Notes |
|-----------|-----------------|----------------|--------------|-------|
| `Dice_Count` | OpenCaptchaWorld latest additions | 9 | 20 | Key-new relative to local cleaned ground truth. |
| `Click_Order` | OpenCaptchaWorld latest additions | 10 | 20 | Key-new relative to local cleaned ground truth. |
| `Patch_Select` | OpenCaptchaWorld latest additions | 10 | 20 | Key-new relative to local cleaned ground truth. |
| `Geometry_Click` | OpenCaptchaWorld latest additions | 11 | 20 | Key-new relative to local cleaned ground truth; all 11 have exact local image hash matches and are warning-annotated, not rejected. |
| `Symbol_Count` | NextGen `Occluded_Pattern_Counting` | 30 | 50 | Normalized to the `metadata.target1_*` single-target count. |
| `Relation_Match` | VIPER `geetest` | 30 | 1000 | Materialized as offline scene image plus object-crop options with `correct_index`. |
| `Hole_Counting` | NextGen `Hole_Counting` | 20 | 20 | Full category included; reconstructed grid composites from source cells. |

Keep NextGen `Layered_Stack`, `Illusory_Ribbons`, `Subway_Paths`, `Rotation_Match`, `Backmost_Layer`, and `Shadow_Direction`, plus VIPER `netease` and `VTT_dataset`, as the first backup pool if later reviewer-driven scope expands beyond the staged Phase 04.2 categories.
