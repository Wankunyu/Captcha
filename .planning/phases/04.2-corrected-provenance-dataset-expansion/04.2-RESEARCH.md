## Implementation Decisions

### Sample Source Strategy

- **D-01:** Prioritize real external CAPTCHA samples from peer-reviewed paper datasets or open-source CAPTCHA datasets.
- **D-02:** Use GPT Image generated Open CaptchaWorld-style samples only as a per-category fallback when real external samples are unavailable, license-unclear, or insufficient for that target category.
- **D-03:** Target at least 10 samples per corrected target category and keep category sample counts as balanced as feasible within the deadline.
- **D-04:** GPT Image fallback rows may be used as direct corrected expanded evidence only with strong provenance caveats. They must not be described as real external dataset evidence or as the primary answer to larger-benchmark concerns.

### Handling Phase 04.1 Artifacts

- **D-05:** Build a new `phase04_2` corrected sidecar and new Phase 04.2 run IDs. Do not overwrite `expanded_captcha_data/phase04_1` or Phase 04.1 run outputs.
- **D-06:** Completely ignore old Phase 04.1 static/adaptive outputs in Phase 04.2 corrected outputs. Phase 04.2 outputs should not compare against, read from, or cite old Phase 04.1 run artifacts.
- **D-07:** Implement Phase 04.2 with separate modules such as `phase042_artifacts.py` and `expanded_dataset_phase042.py`, copied/adapted from Phase 04.1 patterns rather than mutating Phase 04.1 modules.
- **D-08:** Phase 04.2 tools must fail closed if selected manifests, preflight matrices, run IDs, or paper-output inputs reference `phase04_1` paths or `phase04_1` run IDs.

### Novelty and Provenance Validation

- **D-09:** Prove novelty against the current local `captcha_data/` corpus using exact image hash deduplication. Exact hash matches against `captcha_data/` are hard failures.
- **D-10:** Use perceptual hash near-duplicate checks as warning/review signals, not automatic hard failures.
- **D-11:** Use source-specific provenance gates: real external samples require source kind, source citation, source license, provenance notes, and novelty hash report; GPT Image samples require generation prompt, model/date, Open CaptchaWorld-style rationale, source/provenance notes, and novelty hash report.
- **D-12:** Rows that fail novelty or provenance validation should be kept only in validation reports. They must not enter the selected manifest consumed by provider preflight or execution.

### New Category Contract Policy

- **D-13:** Prefer keeping `Symbol_Count` and `Relation_Match` as the two new static-compatible categories, replacing or regenerating their samples with compliant provenance.
- **D-14:** If compliant samples cannot be found for `Symbol_Count` or `Relation_Match`, replacement categories require user confirmation before planning or implementation proceeds.
- **D-15:** Prefer changing only samples, metadata, and provenance for new categories. Keep prompt/schema/scoring contracts stable where feasible; make only small contract adjustments required by compliant samples.
- **D-16:** If GPT Image fallback is needed, save final samples and manifest notes. Prefer using available Codex/GPT image generation directly; if unavailable, prepare style-consistent prompts for web generation. Maintain visual style consistency across generated samples.

### Paid Rerun Gate Sequence

- **D-17:** Execute Phase 04.2 through staged gates: source triage, provenance/novelty validator, selected manifest/materialization, static preflight, static paid runs, adaptive preflight, adaptive paid runs, analysis, and paper-output regeneration.
- **D-18:** Static corrected supplemental runs must cover the full existing paper-facing model matrix, consistent with Phase 04.1 and main paper evidence.
- **D-19:** Adaptive corrected supplemental runs must cover all corrected adaptive-eligible rows, after adaptive preflight and cost gating.
- **D-20:** Static paid runs may continue after successful static preflight. Adaptive paid runs require user confirmation after reviewing adaptive request counts, approximate cost, and output directories.

### Pattern 3: Exact Hash Novelty Hard Gate

**What:** Build a SHA-256 byte-hash index of all current `captcha_data` image files and reject any candidate image whose SHA-256 appears in that index. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`; hash audit command]

```python
IMAGE_SUFFIXES = {".bmp", ".gif", ".jpeg", ".jpg", ".png", ".webp"}

def sha256_image(path: Path) -> str:
    return hashlib.sha256(path.read_bytes()).hexdigest()

def build_captcha_data_hash_index(captcha_root: Path) -> dict[str, list[str]]:
    index: dict[str, list[str]] = {}
    for path in captcha_root.rglob("*"):
        if path.is_file() and path.suffix.lower() in IMAGE_SUFFIXES:
            index.setdefault(sha256_image(path), []).append(str(path))
    return index
```

```python
def average_hash(path: Path, size: int = 8) -> int:
    with Image.open(path) as image:
        gray = image.convert("L").resize((size, size))
        pixels = list(gray.getdata())
    threshold = sum(pixels) / len(pixels)
    bits = 0
    for index, pixel in enumerate(pixels):
        if pixel >= threshold:
            bits |= 1 << index
    return bits

def hamming_distance(left: int, right: int) -> int:
    return (left ^ right).bit_count()
```

## Common Pitfalls

### Pitfall 1: Phase 04.1 Evidence Leakage

**What goes wrong:** A corrected output accidentally reads `expanded_captcha_data/phase04_1`, `phase04_1_static_supplemental`, `phase04_1_adaptive_supplemental`, or `phase04_1_paper_outputs`. [VERIFIED: `results/revision`; `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Why it happens:** Phase 04.1 CLI defaults and test fixtures are hard-coded with `phase04_1` paths/run IDs. [VERIFIED: `expanded_dataset.py`; `tests/test_phase041_*`] 

**How to avoid:** Define Phase 04.2 constants once and use `assert_no_phase041_reference(...)` across loaders, models, CLI args, matrix readers, and paper-output gates. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Warning signs:** Any generated Phase 04.2 CSV/JSON/Markdown contains `phase04_1`, `synthetic_fixture`, `phase04_1-symbol-count`, `phase04_1-relation-match`, or "Copied from existing local captcha_data". [VERIFIED: `expanded_captcha_data/phase04_1/manifest.json`; `results/revision/phase04_1_paper_outputs`]

### Pitfall 2: False Novelty from Renamed Files

**What goes wrong:** A row looks new because it lives under `expanded_captcha_data/phase04_2`, but its image bytes are identical to current `captcha_data`. [VERIFIED: hash audit command]

**Why it happens:** Phase 04.1 copied old data into sidecar paths; local audit found exact duplicates for four supplemented categories. [VERIFIED: `expanded_captcha_data/phase04_1/manifest.json`; hash audit command]

**How to avoid:** Hash image bytes, not filenames or source notes, and reject exact SHA-256 matches before writing selected manifest. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Warning signs:** Candidate source paths resemble old task filenames such as `dice*.png`, `image*.jpg`, `order*.png`, or vendor-prefixed Geometry_Click files already present in `captcha_data`. [VERIFIED: `rg --files captcha_data`; `rg --files expanded_captcha_data/phase04_1`]

### Pitfall 3: Perceptual Hash Overreach

**What goes wrong:** A visually similar but valid external/GPT Image sample is rejected because perceptual hash distance is small. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Why it happens:** Perceptual hashing is approximate and can flag style-consistent images that are not byte duplicates. [ASSUMED]

**How to avoid:** Treat perceptual hashes as `review_warning` values only; exact SHA-256 is the only novelty hard gate. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Warning signs:** Validation code rejects a candidate when `exact_hash_match=false` but `perceptual_near_match=true`. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

### Pitfall 4: Adaptive Scope Narrowing

**What goes wrong:** Adaptive supplemental runs cover only a manually requested subset instead of all corrected adaptive-eligible selected rows/task types. [VERIFIED: `expanded_dataset.py`; `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Why it happens:** Phase 04.1 `build_adaptive_preflight_matrix` allows `requested_task_types`, but Phase 04.2 D-19 requires all corrected adaptive-eligible rows. [VERIFIED: `expanded_dataset.py`; `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**How to avoid:** The paper-facing Phase 04.2 adaptive command should default to all selected manifest rows where `adaptive_eligible=True`; if a subset option remains for debugging, generated paper evidence must reject subset matrices. [VERIFIED: `expanded_dataset.py`; `.planning/REQUIREMENTS.md`]

**Warning signs:** Adaptive preflight matrix task list is shorter than the selected manifest's adaptive-eligible task list. [VERIFIED: `expanded_dataset.py`]

### Pitfall 5: Paid Adaptive Run Without Confirmation

**What goes wrong:** Adaptive collection starts immediately after preflight and spends provider budget before the user reviews cost/counts/output dirs. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Why it happens:** Phase 04.1 `collect-adaptive` does not require a separate confirmation flag. [VERIFIED: `expanded_dataset.py`]

**How to avoid:** Add `--confirmed-adaptive-cost` or a checkpoint-approved confirmation file requirement to Phase 04.2 adaptive collection. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Warning signs:** `collect-adaptive` can run from only a preflight matrix path and secrets file. [VERIFIED: `expanded_dataset.py`]

### Pitfall 6: Paper Outputs Contain Prototype Rows

**What goes wrong:** Paper-ready tables cite `Symbol_Count` or `Relation_Match` rows that came from local scripted Phase 04.1 fixtures. [VERIFIED: `expanded_captcha_data/phase04_1/manifest.json`; `results/revision/phase04_1_paper_outputs`]

**Why it happens:** Phase 04.1 generated paper outputs before the corrected provenance definition was inserted. [VERIFIED: `.planning/STATE.md`; `.planning/ROADMAP.md`]

**How to avoid:** Paper-output generation must read only Phase 04.2 selected manifest and corrected summaries, then scan generated outputs for invalid tokens before reporting success. [VERIFIED: `.planning/phases/04.2-corrected-provenance-dataset-expansion/04.2-CONTEXT.md`]

**Warning signs:** Generated paper files contain `synthetic_fixture`, "Scripted local prototype", `phase04_1-symbol-count`, or `phase04_1-relation-match`. [VERIFIED: `expanded_captcha_data/phase04_1/manifest.json`]
