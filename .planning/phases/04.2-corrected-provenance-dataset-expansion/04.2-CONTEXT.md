# Phase 04.2: Corrected Provenance Dataset Expansion - Context

**Gathered:** 2026-05-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 04.2 rebuilds the Phase 04.1 expanded sidecar under the corrected provenance and novelty definition, then regenerates supplemental static/adaptive evidence and paper-facing outputs from paper-eligible rows only.

This phase corrects the evidence chain for expanded-dataset claims. It must remain offline and dataset-based, must not mutate the original `captcha_data/` corpus, must not run live CAPTCHA automation, and must not use old Phase 04.1 copied/prototype rows as corrected direct evidence.

</domain>

<decisions>
## Implementation Decisions

### Sample Source Strategy

- **D-01:** Prioritize real external CAPTCHA samples from peer-reviewed paper datasets or open-source CAPTCHA datasets.
- **D-02:** Use GPT Image generated Open CaptchaWorld-style samples only as a per-category fallback when real external samples are unavailable, license-unclear, or insufficient for that target category.
- **D-03:** Target at least 10 samples per corrected target category and keep category sample counts as balanced as feasible within the deadline.
- **D-04:** GPT Image fallback rows may be used as direct corrected expanded evidence only with strong provenance caveats. They must not be described as real external dataset evidence or as the primary answer to larger-benchmark concerns.

### Handling Phase 04.1 Artifacts

- **D-05:** Build a new `phase04_2` corrected sidecar and new Phase 04.2 run IDs. Do not overwrite `expanded_captcha_data/phase04_1` or Phase 04.1 run outputs.
- **D-06:** Completely ignore old Phase 04.1 static/adaptive outputs in Phase 04.2 corrected outputs. Phase 04.2 outputs should not compare against, read from, or cite old Phase 04.1 run artifacts.
- **D-07:** Implement Phase 04.2 with separate modules such as `phase042_artifacts.py` and `expanded_dataset_phase042.py`, copied/adapted from Phase 04.1 patterns rather than mutating Phase 04.1 modules.
- **D-08:** Phase 04.2 tools must fail closed if selected manifests, preflight matrices, run IDs, or paper-output inputs reference `phase04_1` paths or `phase04_1` run IDs.

### Novelty and Provenance Validation

- **D-09:** Prove novelty against the current local `captcha_data/` corpus using exact image hash deduplication. Exact hash matches against `captcha_data/` are hard failures.
- **D-10:** Use perceptual hash near-duplicate checks as warning/review signals, not automatic hard failures.
- **D-11:** Use source-specific provenance gates: real external samples require source kind, source citation, source license, provenance notes, and novelty hash report; GPT Image samples require generation prompt, model/date, Open CaptchaWorld-style rationale, source/provenance notes, and novelty hash report.
- **D-12:** Rows that fail novelty or provenance validation should be kept only in validation reports. They must not enter the selected manifest consumed by provider preflight or execution.

### New Category Contract Policy

- **D-13:** Prefer keeping `Symbol_Count` and `Relation_Match` as the two new static-compatible categories, replacing or regenerating their samples with compliant provenance.
- **D-14:** If compliant samples cannot be found for `Symbol_Count` or `Relation_Match`, replacement categories require user confirmation before planning or implementation proceeds.
- **D-15:** Prefer changing only samples, metadata, and provenance for new categories. Keep prompt/schema/scoring contracts stable where feasible; make only small contract adjustments required by compliant samples.
- **D-16:** If GPT Image fallback is needed, save final samples and manifest notes. Prefer using available Codex/GPT image generation directly; if unavailable, prepare style-consistent prompts for web generation. Maintain visual style consistency across generated samples.

### Paid Rerun Gate Sequence

- **D-17:** Execute Phase 04.2 through staged gates: source triage, provenance/novelty validator, selected manifest/materialization, static preflight, static paid runs, adaptive preflight, adaptive paid runs, analysis, and paper-output regeneration.
- **D-18:** Static corrected supplemental runs must cover the full existing paper-facing model matrix, consistent with Phase 04.1 and main paper evidence.
- **D-19:** Adaptive corrected supplemental runs must cover all corrected adaptive-eligible rows, after adaptive preflight and cost gating.
- **D-20:** Static paid runs may continue after successful static preflight. Adaptive paid runs require user confirmation after reviewing adaptive request counts, approximate cost, and output directories.

### the agent's Discretion

- The planner may choose exact Phase 04.2 class names, helper names, and output filenames, provided the separate Phase 04.2 module and path boundaries above are preserved.
- The planner may decide whether to implement perceptual hashing with a small local helper or an available dependency, provided exact hash novelty remains the hard gate.
- The planner may choose the exact validation-report schema shape, provided rejected rows cannot enter the selected manifest.
- The planner may choose whether to keep GPT Image prompt packs as Markdown, JSON, or both, provided final manifest notes and visual consistency requirements are captured.

</decisions>

### Established Patterns

- Revision outputs live under `results/revision/<run_id>/`.
- Sidecar evaluator data should materialize as `<dataset_root>/<TaskType>/ground_truth.json`.
- Task type remains the primary comparison unit; task family and structural tags are explanatory.
- Provider/runtime/protocol failures remain separate from scientific wrong answers.
- The 40% cutoff is a reporting heuristic, not a universal CAPTCHA security boundary.
- Secret values from `secrets.yaml` must never appear in planning docs, reports, or shareable artifacts.

### Integration Points

- Phase 04.2 selected manifests and materialized data should live under `expanded_captcha_data/phase04_2/`.
- Phase 04.2 static run IDs should use a `phase04_2_static_supplemental` prefix.
- Phase 04.2 adaptive run IDs should use a `phase04_2_adaptive_supplemental` prefix.
- Phase 04.2 paper outputs should use a `phase04_2_paper_outputs` run ID and must reject invalid Phase 04.1 prototype rows.
